Проект:
- TypeScript 5.x.
- Чистая архитектура: `domain`, `application`, `infrastructure`, `presentation`.
- Сборка через SWC и Rollup.
- Результат сборки должен быть одним browser-loadable JavaScript-файлом.
- Деплойный `dist/index.js` должен быть максимально минифицирован; не публиковать читаемый bundle на GitHub Pages.
- Деплой конечного JavaScript-файла выполняется через GitHub Actions на GitHub Pages.

Код:
- Спагетти-код неприемлем: разбивай код на небольшие модули и функции с одной понятной ответственностью.
- Не смешивай доменную логику, сценарии приложения, инфраструктуру и UI-детали в одном месте.
- Рецепты профессий держи отдельной доменной моделью, связанной с каноническим `BotResource` по `resourceId`; цвета и уровень в UI/логах бери из связанного ресурса.
- Сценарии добычи и крафта должны запускаться независимо и сообщать UI о ходе работы через события use case, не управляя DOM напрямую.

Цель продукта:
- Делаем простого бота для браузерной игры.
- Бот должен помогать собирать ресурсы через сканирование локации на мобов и ресурсы.
- Реализуем функциональность пошагово, без преждевременного усложнения.

Текущий UI-ориентир:
- При загрузке скрипта на страницу показывать круглую кнопку с иконкой кирки.
- Кнопка с киркой должна быть перетаскиваемой и сохранять позицию между перезапусками через локальное браузерное хранилище.
- Размер окна бота должен быть изменяемым пользователем и сохраняться между перезапусками через локальное браузерное хранилище.
- Для текущих длительных процессов внизу панели показывать process bar: видимая длительность процесса и прогресс, сам бар высотой около 10px.
- Таймеры process bar должны синхронизироваться с событиями use case; добыча уровня 0 считается от успешного старта и не должна растягиваться временем контрольных HTTP-сканов.
- Стиль интерфейса: темная тема, лаконично, аккуратно, с ощущением игрового инструмента.
- По клику на кнопку открывать небольшое окно интерфейса бота рядом с кнопкой.
- В окне нужны блок логов, кнопка начала добычи и мультиселект ресурсов.
- В панели должна быть возможность полностью очистить текущий лог без добавления новой записи об очистке.
- В панели должен быть одиночный селект рабочей локации. Текущие локации: `Селение Баурвилл` с `area_id=6` и `Королевские усыпальницы` с `area_id=157`; выбранную локацию сохранять в локальное браузерное хранилище.
- Формат лога: `время: сообщение`.
- Окно должно закрываться крестиком и быть перетаскиваемым.
- Ресурсы должны иметь постоянные id в модели бота; текущая локальная база: агат, аквамарин, бирюза.
- Ресурсы должны иметь уровень; в UI и логах ресурс показывать в формате `Название [<level>]`.
- Длительность добычи хранится в модели каждого канонического ресурса; текущие агат, аквамарин и бирюза добываются по `20` секунд.
- Текущие рецепты профессий: `Пыль агата` (`recipe_id=51536385`, ресурс `agate`), `Пыль аквамарина` (`recipe_id=51536386`, ресурс `aquamarine`), `Пыль бирюзы` (`recipe_id=51536387`, ресурс `turquoise`).
- Ресурсы из ответа `<farm><item>` не являются отдельной моделью ресурса: это найденные экземпляры канонического `BotResource`, связанные через серверный `artikul_id`.
- При переносе curl-запросов в код брать только URL и HTTP-метод; cookies, session-токены и браузерные заголовки не сохранять в репозитории.
- Неожиданный ответ сервера на HTTP/XML-запросы считать аварийным состоянием: останавливать соответствующий процесс, при включенном alarm-toggle запускать звуковую сирену и всегда писать в лог красную надпись `ТРЕБУЕТСЯ УЧАСТИЕ ЧЕЛОВЕКА`.
- В панели держать toggle сирены; выключенная сирена не проигрывает звук ни при аварии, ни при ручной проверке, но красная строка участия человека при аварии остается обязательной.
- Координаты мобов и ресурсов из сканирования храни в модели/сторе последнего скана, но не выводи в UI без отдельной задачи.
- Для базовой стратегии добычи считать опасными мобов с `agrlevel > 0`; ресурс безопасен, если в радиусе 100 пикселей от него нет опасного моба.
- Цикл добычи: сканировать выбранную зону, выбрать самый безопасный ресурс из выбранных, отправить старт добычи, затем в течение времени добычи ресурса контролировать безопасность повторным сканом каждые 5 секунд.
- После успешного завершения добычи делать паузу 2 секунды перед следующей добычей.
- Если безопасного ресурса нет, ждать 20 секунд и повторять цикл; отмена добычи выполняется запросом `https://w1.dwar.ru/hunt_conf.php?mode=farm&action=cancel`.
- На каждое начало сбора ресурса писать ровно один лог вида `Начата добыча <resource>`, где ресурс выводится цветным тэгом по аналогии с тэгами мобов.
- Не выводить пользователю проценты спокойности/безопасности и логи контрольных проверок; контроль опасности нужен только для экстренной остановки добычи.
- Выбор ресурсов в мультиселекте сохранять в локальное браузерное хранилище.
- Запущенный цикл добычи должен перечитывать текущий выбор ресурсов перед каждой новой итерацией выбора цели, а не кэшировать список на весь запуск.
- Для каждого запроса старта добычи генерировать новый 32-символьный hex `sig`.
- После старта добычи читать XML-ответ `<req>`; если `first_farmer` не равен `1`, сразу отменять добычу и не считать старт успешным.
- Крафт рецептов выполняется отдельными задачами параллельно добыче: для каждого выбранного рецепта отправлять POST на `service_craft.php` с `amount`, максимум `10` за запрос; cooldown считать как `30 секунд * amount`, пауза между запросами крафта одного рецепта `5 секунд`.
- Количество крафта задается одним общим UI-контролом `amount`, по умолчанию `10`; запущенный цикл крафта перечитывает значение перед каждым запросом.
- Стоп добычи/крафта во время активного действия должен сразу визуально вернуть кнопку в состояние старта, дождаться завершения текущего progress bar и только затем остановить цикл; стоп во время паузы должен прерывать паузу мгновенно.
- Если после stop во время активной добычи пользователь снова нажал старт до завершения текущего progress bar, stop-запрос отменяется и текущий цикл добычи продолжает работу.
- Если после stop во время активного крафта пользователь снова нажал старт до завершения текущих cooldown progress bar, продолжение ставится в очередь и новый цикл крафта стартует только после завершения старых активных откатов.
